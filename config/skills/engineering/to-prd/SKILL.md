此技能將當前對話上下文和程式碼理解合成為一份 PRD。不要訪談使用者——直接綜合你已知的內容。

## 流程

1. 探索 repo 了解程式碼庫的當前狀態（如果尚未完成）。整個 PRD 使用專案的 `CONTEXT.md` 領域詞彙，並尊重相關的 ADR。

2. 勾勒出你需要建立或修改的主要模組，尋找可獨立測試的深層模組（deep modules）。

   深層模組（相對於淺層模組）將大量功能封裝在一個簡單、可測試的介面中，且該介面很少變更。

   與使用者確認這些模組是否符合預期、哪些模組需要撰寫測試。

3. 使用下方模板撰寫 PRD，然後發布到專案 issue tracker。標記 `ready-for-agent` 標籤——無需額外分類。

<prd-template>

## 問題陳述 (Problem Statement)

使用者面臨的問題，從使用者的視角出發。

## 解決方案 (Solution)

問題的解決方案，從使用者的視角出發。

## 使用者故事 (User Stories)

一份冗長、編號的使用者故事清單。每個故事的格式為：

1. 作為 <角色>，我想要 <功能>，以便 <收益>

<user-story-example>
1. 作為行動銀行客戶，我想要查看帳戶餘額，以便做出更好的消費決策
</user-story-example>

使用者故事應極其廣泛，涵蓋功能的所有面向。

## 實作決策 (Implementation Decisions)

已做出的實作決策清單，可包含：

- 將建立/修改的模組
- 將修改的模組介面
- 開發者的技術澄清
- 架構決策
- Schema 變更
- API 合約
- 具體的互動行為

不要包含具體的檔案路徑或程式碼片段——它們可能很快就會過時。

例外：如果原型產生了一段比散文更精確編碼決策的片段（狀態機、reducer、schema、型別形狀），則在相關決策中內聯並簡要說明它來自原型。只保留富含決策的部分——不是可運行的 demo，而是關鍵部分。

## 測試決策 (Testing Decisions)

已做出的測試決策清單。包含：

- 什麼構成一個好的測試的描述（只測試外部行為，不測試實作細節）
- 哪些模組將被測試
- 測試的參考先例（即程式碼庫中類似類型的測試）

## 不在此範圍 (Out of Scope)

此 PRD 範圍之外的事項。

## 補充說明 (Further Notes)

關於此功能的任何補充說明。

</prd-template>
