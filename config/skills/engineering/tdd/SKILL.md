# 測試驅動開發 (TDD)

## 哲學

**核心原則**：測試應透過公開介面驗證行為，而非實作細節。程式碼可以完全改變；測試不應該。

**好的測試**是整合風格的：它們透過公開 API 執行真實的程式碼路徑。它們描述系統做什麼，而非它如何做。好的測試讀起來像規格——「使用者可以用有效購物車結帳」精確地告訴你存在什麼能力。這些測試在重構中存活，因為它們不在乎內部結構。

**壞的測試**與實作耦合。它們 mock 內部協作者、測試私有方法或透過外部手段驗證（如直接查詢資料庫而非使用介面）。警告信號：當你重構時測試失敗，但行為沒有改變。如果你重新命名一個內部函數而測試失敗，那些測試是在測試實作，而非行為。

## 反模式：水平切片

**不要先寫所有測試，再寫所有實作。** 這是「水平切片」——將 RED 視為「寫所有測試」，將 GREEN 視為「寫所有程式碼」。

這會產生 **劣質測試**：

- 批量編寫的測試測試的是想像中的行為，而非實際行為
- 你最終測試的是事物的形狀（資料結構、函數簽章）而非面向使用者的行為
- 測試對真實變更不敏感——行為崩壞時它們通過，行為正常時它們失敗
- 你跑得比自己的車頭燈還遠，在理解實作之前就承諾了測試結構

**正確方法**：透過 tracer bullets 的垂直切片。一個測試 → 一個實作 → 重複。每個測試回應你從上一個循環中學到的東西。因為你剛寫完程式碼，你確切知道什麼行為重要以及如何驗證它。

```
錯誤（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正確（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流程

### 1. 規劃

探索程式碼庫時，使用 `CONTEXT.md` 領域詞彙使測試名稱和介面詞彙與專案語言一致，並尊重相關 ADR。

編寫任何程式碼之前：

- [ ] 與使用者確認需要哪些介面變更
- [ ] 與使用者確認要測試哪些行為（排優先級）
- [ ] 識別深層模組（小介面、深實作）的機會
- [ ] 為可測試性設計介面
- [ ] 列出要測試的行為（非實作步驟）
- [ ] 取得使用者對計劃的批准

提問：「公開介面應該是什麼樣子？哪些行為最重要需要測試？」

**你無法測試一切。** 與使用者確切確認哪些行為最重要。將測試努力集中在關鍵路徑和複雜邏輯上，而非每個可能的邊界情況。

### 2. Tracer Bullet

撰寫一個測試，確認系統的一件事：

```
RED:   為第一個行為寫測試 → 測試失敗
GREEN: 寫最小程式碼使其通過 → 測試通過
```

這是你的 tracer bullet——證明路徑端到端可用。

### 3. 增量循環

對每個剩餘行為：

```
RED:   寫下一個測試 → 失敗
GREEN: 寫最小程式碼使其通過 → 通過
```

規則：

- 一次一個測試
- 只寫足夠通過當前測試的程式碼
- 不要預先考慮未來的測試
- 保持測試專注於可觀察的行為

### 4. 重構

所有測試通過後，尋找重構候選：

- [ ] 提取重複
- [ ] 深化模組（將複雜性移到簡單介面背後）
- [ ] 在自然的地方應用 SOLID 原則
- [ ] 考慮新程式碼揭示了現有程式碼的什麼
- [ ] 每個重構步驟後執行測試

**永遠不要在 RED 時重構。** 先到 GREEN。

## 每個循環的檢查清單

```
[ ] 測試描述行為，而非實作
[ ] 測試僅使用公開介面
[ ] 測試在內部重構後仍能存活
[ ] 程式碼對這個測試來說是最小的
[ ] 未添加推測性功能
```
