# To Issues

使用垂直切片（tracer bullets）將計劃分解為可獨立抓取的 issues。

## 流程

### 1. 收集上下文

從對話上下文中已有的內容開始。如果使用者傳遞了 issue 參考（issue 編號、URL 或路徑）作為參數，從 issue tracker 取得並閱讀其完整內容和評論。

### 2. 探索程式碼庫（可選）

如果你尚未探索過程式碼庫，請了解程式碼的當前狀態。Issue 標題和描述應使用 `CONTEXT.md` 中的領域詞彙，並尊重相關的 ADR。

### 3. 草擬垂直切片

將計劃分解為 **tracer bullet** issues。每個 issue 是一個薄的垂直切片，端到端穿透所有整合層，而不是某一層的水平切片。

切片可以是 'HITL' 或 'AFK'。HITL 切片需要人機互動，如架構決策或設計審查。AFK 切片可以在沒有人機互動的情況下實作和合併。盡可能選擇 AFK 而非 HITL。

<vertical-slice-rules>
- 每個切片交付一條狹窄但完整的路徑，穿透每一層（schema、API、UI、測試）
- 一個完成的切片可以獨立示範或驗證
- 偏好許多薄切片而非少數厚切片
</vertical-slice-rules>

### 4. 詢問使用者

以編號清單呈現建議的分解。對每個切片顯示：

- **標題**：簡短描述性名稱
- **類型**：HITL / AFK
- **阻塞於**：哪些其他切片（如有）必須先完成
- **涵蓋的使用者故事**：此切片處理哪些使用者故事（如果來源有）

詢問使用者：

- 粒度是否合適（太粗 / 太細）？
- 依賴關係是否正確？
- 是否有切片應合併或進一步拆分？
- 是否正確標記了 HITL 和 AFK？

迭代直到使用者批准分解。

### 5. 將 issues 發布到 issue tracker

對每個批准的切片，發布新 issue 到 issue tracker。使用下方的 issue 主體模板。這些 issues 被視為 AFK agent 可立即處理，因此使用正確的分類標籤發布，除非另有指示。

按依賴順序發布 issues（阻塞者優先），這樣可以在「阻塞於」欄位中引用真實的 issue 識別碼。

<issue-template>
## 父級 (Parent)

對 issue tracker 上父級 issue 的參考（如果來源是現有 issue，否則省略此節）。

## 要建立的內容 (What to build)

此垂直切片的簡潔描述。描述端到端的行為，而非逐層實作。

避免具體的檔案路徑或程式碼片段——它們很快就會過時。例外：如果原型產生了一個比散文更精確編碼決策的片段（狀態機、reducer、schema、型別形狀），在此內聯並簡要說明它來自原型。只保留富含決策的部分——不是可運行的 demo，而是關鍵部分。

## 驗收標準 (Acceptance criteria)

- [ ] 標準 1
- [ ] 標準 2
- [ ] 標準 3

## 阻塞於 (Blocked by)

- 對阻塞票證的參考（如有）

或「無 — 可立即開始」，如果沒有阻塞者。

</issue-template>

不要關閉或修改任何父級 issue。
