# 改進程式碼庫架構

揭示架構摩擦並提出 **深化機會 (deepening opportunities)**——將淺層模組變成深層模組的重構。目標是可測試性和 AI 可導航性。

## 詞彙表

在每個建議中精確使用這些術語。一致的語言是重點——不要漂移到「組件」、「服務」、「API」或「邊界」。

- **模組 (Module)**——任何有介面和實作的東西（函數、類別、套件、切片）。
- **介面 (Interface)**——呼叫者使用模組所需知道的一切：型別、不變量、錯誤模式、排序、配置。不僅僅是型別簽章。
- **實作 (Implementation)**——內部的程式碼。
- **深度 (Depth)**——介面處的槓桿：大量行為藏在一個小介面背後。**深層** = 高槓桿。**淺層** = 介面幾乎和實作一樣複雜。
- **接縫 (Seam)**——介面所在之處；可以在不原地編輯的情況下改變行為的地方。（使用此術語，而非「邊界」。）
- **適配器 (Adapter)**——在接縫處滿足介面的具體事物。
- **槓桿 (Leverage)**——呼叫者從深度中獲得的東西。
- **局部性 (Locality)**——維護者從深度中獲得的東西：變更、bug、知識集中在一個地方。

關鍵原則：

- **刪除測試**：想像刪除該模組。如果複雜性消失，它只是一個傳遞層。如果複雜性在 N 個呼叫者中重新出現，它是在賺取它的存在價值。
- **介面是測試表面。**
- **一個適配器 = 假設的接縫。兩個適配器 = 真實的接縫。**

此技能以專案的領域模型為基礎。領域語言為好的接縫命名；ADR 記錄此技能不應重新討論的決策。

## 流程

### 1. 探索

先閱讀 `CONTEXT.md` 和相關 ADR。

然後使用 Agent 工具以 `subagent_type=Explore` 遍歷程式碼庫。不要遵循僵化的啟發法——有機地探索並注意你遇到摩擦的地方：

- 在哪裡理解一個概念需要在許多小模組之間跳轉？
- 哪些模組是**淺層**的——介面幾乎和實作一樣複雜？
- 哪些純函數被提取出來只是為了可測試性，但真正的 bug 藏在它們如何被呼叫中（沒有**局部性**）？
- 哪些緊耦合模組洩漏穿過它們的接縫？
- 程式碼庫的哪些部分未經測試，或難以透過當前介面測試？

對你懷疑是淺層的任何東西應用**刪除測試**：刪除它會集中複雜性，還是僅僅移動它？「是，會集中」是你想要的信號。

### 2. 呈現候選

以編號清單呈現深化機會。對每個候選：

- **檔案**——涉及哪些檔案/模組
- **問題**——為什麼當前架構造成摩擦
- **解決方案**——對什麼將改變的簡明描述
- **好處**——以局部性和槓桿來解釋，以及測試將如何改善

**使用 `CONTEXT.md` 詞彙來討論領域。** 如果 `CONTEXT.md` 定義了某個概念，使用該術語——不要漂移。

**ADR 衝突**：如果候選與現有 ADR 矛盾，僅在摩擦足夠真實以值得重新審視 ADR 時才提出。清楚標記（例如「與 ADR-0007 矛盾——但值得重新開啟因為……」）。不要列出 ADR 禁止的每個理論重構。

不要在此階段提出具體介面。詢問使用者：「你想探索哪一個？」

### 3. 探究循環

一旦使用者選擇了候選，進入探究對話。與使用者一起走設計樹——約束、依賴、深化模組的形狀、接縫背後是什麼、哪些測試存活。

決策結晶時內聯產生副作用：

- **以不在 `CONTEXT.md` 中的概念命名深化模組？** 將術語添加到 `CONTEXT.md`——與 grill-with-docs 相同的紀律。
- **在對話中銳化模糊術語？** 立即更新 `CONTEXT.md`。
- **使用者以有份量的理由拒絕候選？** 提供 ADR，框架為：「想讓我將此記錄為 ADR，以便未來的架構審查不會重新建議嗎？」僅在理由確實需要被未來探索者看到以避免重新建議相同事物時提供——跳過短暫的理由（「現在不值得」）和自明的理由。
